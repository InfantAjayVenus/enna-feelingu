# Module Boundaries

> Generated by /bee:discover from the project's milestone map.
> Edit this file to adjust boundaries as the project evolves.

## Modules

- `app/` — owns: screens, navigation, routing (Expo Router file-based routes). Depends on: features/*, store/
- `features/checkin/` — owns: CheckInForm, SkipAction, AutoSkip, OverrideSkipped, RetroactiveEntry. Depends on: store/
- `features/history/` — owns: EntryList, DayView, MoodChart, WeekView, MonthView. Depends on: store/
- `features/notifications/` — owns: scheduling, permission handling. Depends on: store/
- `features/settings/` — owns: active hours config (with granular hour selection). Depends on: store/
- `features/export/` — owns: CSV serializer, JSON serializer, time-window picker, share-sheet trigger. Depends on: store/
- `store/` — owns: local persistence layer (all entries, settings, custom exercises). Depends on: (none)

## Rules

- Modules may only import from declared dependencies
- Domain concepts live in their owning module — do not scatter across modules
- No circular dependencies
- `app/` screens are thin — they delegate logic to feature modules, not implement it themselves
- `store/` has no knowledge of UI or features — it is a pure data layer
- When in doubt, duplicate code rather than create a coupling that violates boundaries
